# Техническое задание (ТЗ) — Telegram Bot + Telegram Mini App для 3D‑принтеров (Klipper/Moonraker)

## 0) Термины
- **Moonraker** — API‑сервер для Klipper (HTTP + WebSocket JSON‑RPC).  
- **Klipper objects / printer objects** — объекты статуса (toolhead, print_stats, virtual_sdcard, extruder и т.д.), которые Moonraker умеет query/subscribe.   
- **Preset (пресет печати)** — загруженный пользователем `.gcode` + метаданные (имя, пластик, цвет, thumbnail) + ограничения совместимости (модель принтера, сопло, минимальный стол).  
- **Printer Model (модель/группа)** — логическая “группа” принтеров (например `Creality K1`, `Ender 3`). Принтер принадлежит ровно одной модели. Пресет можно разрешить только для выбранных моделей (чтобы K1‑gcode нельзя было запустить на Ender).  
- **Mini App** — фронтенд внутри Telegram (web), размещён на Vercel.  
- **Backend** — ваш локальный сервер рядом с принтерами, который ходит в Moonraker и отдает данные Mini App + шлет уведомления ботом.

---

## 1) Цели продукта (что должно уметь “из коробки”)
1) С телефона **удаленно** видеть:
   - список принтеров, их состояния (printing/paused/error/standby/offline),
   - прогресс, оставшееся время (ETA),
   - температуры,
   - текущий файл,
   - текущий слой / всего слоев (если настроено). 

2) Управление **минимальное**:
   - Pause / Resume / Cancel для текущей печати,
   - запуск пресета на один или несколько принтеров (только совместимые). 

3) В Mini App:
   - добавление/удаление принтеров,
   - при добавлении — указывать имя принтера и модель (K1/Ender3/…), **спеки подтягивать из Moonraker** (область печати, сопло если возможно),
   - библиотека пресетов (gcode) с картинкой (thumbnail), типом пластика, цветом, названием,
   - **зависимости пресетов**: пресет запускается только на принтерах подходящей модели/сопла/размера стола.

4) **История печатей** единым списком (по всем принтерам).

5) **Умные уведомления без спама**:
   - “первый слой готов” + оставшееся время + краткая сводка (принтер/файл/слой),
   - “печать завершена”,
   - “ошибка” с подробностями.

## 2) Размещение и доступ (обязательная часть для “удаленно”)
### 2.1 Компоненты
- **Frontend (Mini App)**: Vercel (HTTPS).
- **Backend + Telegram bot**: локальная сеть рядом с Moonraker.
- **Moonraker**: локальная сеть.

### 2.2 Требование: публичный HTTPS до Backend
Так как Mini App на телефоне в Telegram — это браузер, он должен ходить в backend по **публичному HTTPS** (и WebSocket по WSS). Локальные `localhost/192.168.*` удаленно работать не будут.

**Требование ТЗ:** обеспечить внешний доступ к backend (например Cloudflare Tunnel / reverse proxy / VPN).  
Backend должен быть доступен как:
- `https://<public-backend-domain>/api/...`
- `wss://<public-backend-domain>/ws`

## 3) Безопасность и авторизация
### 3.1 Авторизация Mini App (обязательно)
Mini App получает `Telegram.WebApp.initData`. На backend нужно **валидировать подпись** (hash) по алгоритму Telegram (HMAC‑SHA256 с ключом `WebAppData`) и дополнительно проверять `auth_date` на “не слишком старое”. 

**Требование:** endpoint `POST /api/auth/telegram`:
- вход: `{ initData: string }`
- backend:
  1) парсит querystring,
  2) строит `data_check_string`,
  3) проверяет hash,
  4) проверяет `auth_date` (например не старше 24 часов),
  5) выдает сессию (JWT/cookie).
- дополнительно: whitelist Telegram `user.id` (система персональная).

### 3.2 Доступ к Moonraker
Для “untrusted clients” Moonraker требует JWT или API key; при API key он передается в заголовке `X-Api-Key`. 

**Требование:**
- хранить для каждого принтера `moonraker_url` и `api_key` (в зашифрованном виде либо хотя бы в закрытом хранилище конфигов).
- Moonraker не открывать наружу напрямую; наружу открывать только ваш backend.

## 4) Интеграция с Moonraker — как именно получать статусы и управлять

### 4.1 Протокол realtime (основной)
Backend для каждого принтера держит WebSocket к:
```text
ws://<moonraker-host>:7125/websocket
```
(это primary websocket Moonraker). 

Далее backend:
1) вызывает JSON‑RPC `server.connection.identify` и получает `connection_id` (это современный способ, `server.websocket.id` deprecated). 
2) вызывает `printer.objects.subscribe` и подписывается на нужные objects. 
3) получает дифф‑апдейты через уведомления `notify_status_update`. 

> Moonraker всегда отправляет `params` массивом. 

### 4.2 Какие объекты подписывать (минимум)
Подписка минимум на:
- `print_stats` — state/filename/durations/message + `info.current_layer/total_layer`. 
- `virtual_sdcard` — `progress` 0..1, активность, размер файла. 
- `display_status` — `progress` (если есть M73, иначе fallback на virtual_sdcard.progress по документации). 
- `toolhead` — `axis_minimum/axis_maximum` (для вычисления области печати) + лимиты скорости/ускорений если нужно отображать. 
- `webhooks` — чтобы понимать ready/shutdown и текст причины. (используется в рекомендациях Moonraker по опросу состояния). 
- `extruder`, `heater_bed` — температуры/цели (если объекты есть). (наличие проверять через `/printer/objects/list`). 

### 4.3 Управляющие команды (минимально)
Использовать стандартные Moonraker endpoints:
- Start: `POST /printer/print/start?filename=...` 
- Pause: `POST /printer/print/pause` 
- Resume: `POST /printer/print/resume` 
- Cancel: `POST /printer/print/cancel` 
- (Опционально “красная кнопка” в UI, но это все равно минимально и полезно): `POST /printer/emergency_stop` — мгновенный emergency stop.

## 5) Принтеры: добавление/удаление + спек‑профиль (вместо “групп”)

### 5.1 Сущности
- **PrinterModel**: логическая модель (“Creality K1”, “Ender 3”).  
- **Printer**: конкретный принтер.

### 5.2 Add Printer (флоу в Mini App)
Экран “Добавить принтер”:
1) `Display Name` (пользовательское имя)
2) `Model` (выбор из списка моделей + “создать новую модель”)
3) `Moonraker URL` (например `http://k1-1.local:7125`)
4) `API Key` (строка)
5) кнопка **Test connection**
6) после успешного теста — показать подтянутые спеки и кнопка **Save**

### 5.3 Как автоматически подтягивать спеки принтера
#### 5.3.1 Размер области печати (стол/ходы)
Из `toolhead.axis_minimum` и `toolhead.axis_maximum` можно вычислить:
- `bedX = axis_maximum[0] - axis_minimum[0]`
- `bedY = axis_maximum[1] - axis_minimum[1]`
- `bedZ = axis_maximum[2] - axis_minimum[2]` 

**Требование:** backend при добавлении принтера делает query toolhead и сохраняет bedX/bedY/bedZ в БД.

#### 5.3.2 Диаметр сопла
В Klipper в `[extruder]` параметр `nozzle_diameter` обязателен.   
Moonraker `configfile.settings` содержит распарсенные настройки с дефолтами и типами (но объект может быть большим). 

**Требование:**
- backend делает однократный запрос `configfile.settings` (или точечный query configfile если реализуете аккуратно) и пытается прочитать:
  - `configfile.settings["extruder"]["nozzle_diameter"]`
- если не получилось — UI требует ручной ввод nozzle (0.4/0.6/…).
- в карточке принтера должна быть кнопка “Rescan specs” (переподтянуть bed/nozzle).

### 5.4 Удаление принтера
- Удалить запись из БД.
- Закрыть WS‑соединение коннектора к Moonraker.
- Историю/пресеты **не удалять** (они общие), но в истории помечать что принтер удален.

## 6) Пресеты (gcode библиотека) + thumbnail + авто‑метаданные

### 6.1 Создание пресета
Форма “Новый пресет”:
- Upload `.gcode` (файл)
- `Title` (строка)
- `Plastic type` (строка/enum)
- `Color` (HEX или выбор)
- `Compatibility` (см. раздел 7)

**Требование:** после загрузки файла backend хранит “мастер‑копию” gcode (локальный диск/volume).  

### 6.2 Метаданные gcode через Moonraker (для автозаполнения)
Moonraker умеет:
- получить метаданные: `GET /server/files/metadata?filename=...` (включая `estimated_time`, `nozzle_diameter`, `filament_type`, `filament_name`, `thumbnails`, и т.д.). 
- форсировать перескан: `POST /server/files/metascan?filename=...` 

**Требование реализации (конкретно):**
- При первом деплое gcode на любой принтер:
  1) backend загружает файл на этот принтер (см. 6.4),
  2) вызывает `metascan`,
  3) читает `metadata`,
  4) сохраняет в Preset поля:
     - `estimated_time_sec` (если есть),
     - `gcode_nozzle_diameter` (если есть),
     - `gcode_filament_type` (если есть),
     - `gcode_filament_name` (если есть).

### 6.3 Thumbnail превью
Moonraker умеет:
- `GET /server/files/thumbnails?filename=...` → список thumbnail с `thumbnail_path` относительно `gcodes` root. 
- скачать файл: `GET /server/files/{root}/{filename}`. 

**Требование:**
- Backend выбирает “лучший” thumbnail (максимальная площадь, но не больше заданного лимита, например до 600px по большей стороне).
- Backend скачивает thumbnail и хранит у себя как `preset_thumbnail.png` (или webp).
- Frontend показывает thumbnail в списке пресетов и на карточке пресета.

### 6.4 Печать пресета на принтере (деплой)
Печать в Moonraker происходит по `filename` в папке `gcodes`, значит файл нужно загрузить. 

Moonraker upload:
- `POST /server/files/upload` (multipart/form-data)
- поддерживает `root`, `path`, `checksum`, а также `print=true` чтобы начать печать сразу после загрузки. 

**Требование “безопасной” реализации:**
- Для каждого принтера делать:
  1) upload в поддиректорию, например `tg_presets/<presetId>/...`
  2) после успешного upload — старт печати через `POST /printer/print/start?filename=...`  
  (не использовать `print=true`, чтобы иметь единый контроль ошибок и логирование старта отдельной операцией). 

**Кеширование:**
- backend считает SHA256 “мастер‑копии” и использует `checksum` при upload (Moonraker умеет сверить и вернуть 422 при несовпадении). 
- хранить в БД таблицу `PresetDeployment` (presetId + printerId + remoteFilename + checksum + uploadedAt), чтобы не перезаливать файл при повторных запусках.

## 7) Совместимость пресетов (чтобы K1‑gcode нельзя было запустить на Ender)

### 7.1 Что должно ограничиваться
Пресет имеет правила:
1) **Allowed Models**: список разрешенных PrinterModel (например только `Creality K1`).
2) **Allowed Nozzle Diameters**: список разрешенных сопел (например только `0.4`).
3) **Min bed size**: минимальный стол X/Y (например 220×220).

### 7.2 Где задаётся
В форме “Новый/Редактировать пресет” блок **Compatibility**:
- Multi‑select `Allowed printer models`
- Multi‑select `Allowed nozzle` (0.2/0.4/0.6/0.8 — можно кастом)
- `Min bed X`, `Min bed Y` (числа)

### 7.3 Как применяется
**Правило (обязательное):** проверка совместимости делается:
- на фронте (для UX — скрывать/дизейблить несовместимые),
- и на backend (как “истина”, чтобы нельзя было обойти).

Backend возвращает причины несовместимости, например:
- `MODEL_NOT_ALLOWED`
- `NOZZLE_NOT_ALLOWED`
- `BED_TOO_SMALL`

### 7.4 UI выбора принтеров при старте
На экране запуска пресета:
- список принтеров только **совместимых** и **готовых** (`print_stats.state=standby/complete/cancelled` и `webhooks.ready=true/ready`) — остальные серые с причиной. 
- переключатель “Выбрать все совместимые”
- кнопка START (с подтверждением “печать на 2 принтерах?”)

## 8) Контроль слоёв (для Mini App и уведомлений)

### 8.1 Источник
Moonraker `print_stats.info.total_layer` и `current_layer` существуют, но будут `null`, пока слайсер не будет отправлять `SET_PRINT_STATS_INFO`. 

**Требование UI/Settings:**
- В Mini App Settings показать подсказку “как включить слои” (минимально: добавить `SET_PRINT_STATS_INFO TOTAL_LAYER=...` в start gcode и `SET_PRINT_STATS_INFO CURRENT_LAYER=...` на layer change).
- В UI показывать слои только если значения не null.

## 9) Умные уведомления (без спама): первый слой / завершение / ошибка

### 9.1 Канал уведомлений
Уведомления отправляет Telegram‑бот в личный чат владельцу (chat_id сохраняется после `/start`).

### 9.2 События‑триггеры (ровно 3 типа)
1) **FIRST_LAYER_DONE**
2) **PRINT_COMPLETE**
3) **PRINT_ERROR**

### 9.3 Как детектить “первый слой готов”
Основной способ (если слои настроены):
- когда `print_stats.info.current_layer` становится `2` → значит первый слой завершён (перешли печатать 2‑й). 

Fallback (если слои = null):
- отправить FIRST_LAYER_DONE когда:
  - `virtual_sdcard.progress >= 0.02` (2%) **и**
  - `print_duration >= 120s` **и**
  - событие еще не отправлено для этой печати. 

### 9.4 Как детектить “завершено” и “ошибка”
- `print_stats.state`:
  - `complete` → PRINT_COMPLETE
  - `error` → PRINT_ERROR 

### 9.5 “Подробности ошибки”
В уведомлении PRINT_ERROR обязательно включать:
- `print_stats.message` (если не пусто) 
- статус Klipper readiness через `webhooks` (если printer не ready/в shutdown) — текст причины 
- (опционально, но в рамках “подробностей”): последние N (например 5–10) сообщений `notify_gcode_response` за последние 30 секунд (логировать на backend). 

### 9.6 Антиспам (обязательно)
- На каждую печать хранить `print_session_id` (см. ниже) и флаги:
  - `firstLayerNotified`
  - `completeNotified`
  - `errorNotified`
- Не отправлять одинаковое событие дважды для одной сессии.
- Если принтер “флапает” online/offline — НЕ слать уведомления об этом (в ТЗ не требуется).

### 9.7 Print Session ID (идентификатор текущей печати)
Использовать комбинацию:
- принтерId
- факт старта печати (state=printing)
- filename
- start_time из history events, если доступно

Moonraker шлёт `notify_history_changed` при добавлении/завершении job, с полями `action` и `job` (включая `start_time`, `filename`, `status`).   
**Требование:** при `action=added` создавать `print_session_id` и привязывать к принтеру как “текущая сессия”.

## 10) История печатей (обязательно)
### 10.1 Источник данных
Moonraker History API:
- `GET /server/history/list?...`
- `GET /server/history/job?uid=...`
- `GET /server/history/totals` 

Moonraker realtime событие:
- `notify_history_changed` (added/completed). 

### 10.2 Требование UI
Экран “History” (либо отдельная вкладка, либо внутри Dashboard):
- список последних N событий по всем принтерам:
  - дата/время
  - принтер
  - файл/пресет
  - статус (completed/cancelled/error)
  - длительность (print_duration/total_duration если доступно)
- фильтры: All / Completed / Error

### 10.3 Требование реализации
Backend агрегирует историю по всем принтерам:
- при `notify_history_changed` — обновляет локальную таблицу `PrintHistory` (с минимальным набором полей),
- по запросу фронта отдает пагинированный список.

## 12) Backend API для фронта (контракт)

Базовый префикс: `/api`

### 12.1 Auth
- `POST /auth/telegram`
  - req: `{ initData: string }`
  - resp: `{ token: string, user: { id, username } }`

### 12.2 Printers
- `GET /printers`
  - resp: массив принтеров + текущий snapshot статуса (кеш)  
- `POST /printers`
- `PATCH /printers/:id`
- `DELETE /printers/:id`
- `POST /printers/:id/test` (тест коннекта)
- `POST /printers/:id/rescan` (bed/nozzle)

### 12.3 Управление печатью
- `POST /printers/:id/pause`
- `POST /printers/:id/resume`
- `POST /printers/:id/cancel`

### 12.4 Presets
- `GET /presets`
- `POST /presets` (multipart: file + json fields)
- `GET /presets/:id`
- `PATCH /presets/:id`
- `DELETE /presets/:id`
- `POST /presets/:id/print` body: `{ printerIds: string[] }`

### 12.5 History
- `GET /history?limit=50&offset=0&status=all|completed|error|cancelled`

### 12.6 Realtime
- `GET /snapshot` — все принтеры одним JSON (для первого рендера)
- `GET /ws` (WebSocket) — пуш‑события статусов

## 13) Схема сообщений backend → frontend (WebSocket)
Backend шлет события в формате:
```json
{
  "type": "PRINTER_STATUS",
  "printerId": "p1",
  "ts": 1710000000,
  "payload": {
    "state": "printing",
    "progress": 0.42,
    "etaSec": 3600,
    "filename": "part.gcode",
    "temps": { "extruder": { "t": 215, "target": 215 }, "bed": { "t": 60, "target": 60 } },
    "layers": { "current": 12, "total": 120 }
  }
}
```

Также:
- `type: "HISTORY_EVENT"` при обновлении истории
- `type: "PRESET_UPDATED"` если меняли пресет

## 14) Внутренняя структура backend (обязательно для программиста)

### 14.1 MoonrakerConnector (1 на принтер)
Функции:
- connect WS
- `server.connection.identify` (получить `connection_id`) 
- subscribe `printer.objects.subscribe` на заданные objects 
- принимать `notify_status_update` диффы и обновлять локальный `snapshot` 
- слушать `notify_history_changed` и обновлять историю + print session 
- reconnect с backoff

### 14.2 CompatibilityService
Метод:
- `checkPresetAllowed(preset, printer) -> { allowed: boolean, reasons: [] }`

### 14.3 NotificationService
- подписан на изменения snapshot и history events
- реализует правила FIRST_LAYER_DONE / COMPLETE / ERROR
- хранит дедуп в БД (`NotificationLog`) или в памяти + persistence (рекомендуется БД)

## 17) Acceptance Criteria (проверка готовности)
1) Добавлены 2 принтера K1 и 1 Ender 3.
2) Для каждого принтера в UI отображаются bedX×bedY×bedZ (из toolhead axis min/max) и nozzleDiameter (из configfile/settings или ручной ввод). 
3) Создан пресет “K1 fast” с allowedModels = [K1].
4) При запуске пресета Ender 3 не доступен и показана причина “несовместимая модель”.
5) Пресет запускается на 2 K1, оба получают upload и старт печати.
6) При переходе на layer 2 приходит уведомление “первый слой готов + ETA”.
7) При `print_stats.state=complete` приходит “завершено”. 
8) При `print_stats.state=error` приходит “ошибка” + `print_stats.message` + доп. информация. 
9) Thumbnail отображается в списке пресетов (из Moonraker thumbnails). 
10) History показывает завершенные/ошибочные печати единым списком (на основе `/server/history/list` и `notify_history_changed`).

## 18) Примечания по источникам данных (чтобы программист не “плавал”)
- Состояние печати и слои: `print_stats` + `SET_PRINT_STATS_INFO` (слои будут null без слайсера). 
- Прогресс: `display_status.progress` (если M73) иначе Moonraker описывает fallback на `virtual_sdcard.progress`. 
- История: `notify_history_changed` + `/server/history/list`. 
- Метаданные/thumbnail: `/server/files/metadata`, `/server/files/metascan`, `/server/files/thumbnails`. 
- Управление печатью: `/printer/print/*` и `/printer/emergency_stop`. 
- Авторизация Mini App: обязательная проверка `initData` на backend по алгоритму Telegram.
